# `product_db` 综合评测题库

## 目标

这份题库用于评测 `data-agent-backend` 的精简版数据分析 Agent，重点覆盖以下能力：

- 意图识别
- 证据召回 / Schema 召回
- 查询增强
- SQL 生成
- SQL 安全控制
- SQL 执行与结果总结
- 防幻觉 / 防注入 / 错误恢复

## 使用方式

建议每道题按以下维度记录结果：

- 是否识别为数据分析问题
- 是否召回了正确表
- SQL 是否可执行
- SQL 语义是否正确
- 结果是否合理
- 是否存在幻觉 / 越权 / 格式漂移

建议判分方式：

- `0`：失败
- `1`：部分正确
- `2`：正确

推荐记录字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 题目编号 |
| `category` | 题型类别 |
| `question` | 用户问题 |
| `difficulty` | 难度：L1/L2/L3/L4 |
| `target_tables` | 预期涉及表 |
| `risk_focus` | 这题重点测试什么 |
| `notes` | 口径说明 / 允许的等价写法 |

## 评测系统 TODO

结合 `Hello-Agents` 第 12 章“智能体性能评估”的思路，当前项目后续建议把评测能力做成独立系统，而不是只靠手工跑题。

### 1. 评测模块分层

- [ ] 新增 `dataset` 层：保存题目、预期表、风险点、标准校验 SQL
- [ ] 新增 `evaluator` 层：负责执行整条评测流程
- [ ] 新增 `metrics` 层：统一计算准确率、可执行率、拦截率等指标
- [ ] 新增 `report` 层：输出 `json/markdown` 评测报告

建议目录形态：

```text
eval/
├── dataset/
├── evaluator/
├── metrics/
└── report/
```

### 2. 第一版核心指标

- [ ] 意图识别准确率：是否正确区分数据分析 / 闲聊
- [ ] 表召回命中率：是否召回预期表
- [ ] SQL 可执行率：生成 SQL 是否能成功执行
- [ ] 结果正确率：执行结果是否与标准校验 SQL 一致
- [ ] 安全拦截率：注入 / 越权 / 危险 SQL 是否被正确拒绝

### 3. 第一版评测方式

优先采用低成本、可自动化的方式：

- [ ] 规则评测：检查 stage 顺序、关键 state 字段、SQL 安全规则
- [ ] 结果比对评测：对功能题执行“生成 SQL”与“标准校验 SQL”，比较结果集
- [ ] 错误恢复评测：验证失败时是否 graceful fail，而不是整条流崩溃

### 4. 第二版增强评测

在第一版稳定后，再增加更高成本但更灵活的评测：

- [ ] LLM Judge：评估结果总结、拒答说明、解释质量
- [ ] Win Rate：对比 prompt / recall / SQL 生成策略改造前后的回答质量
- [ ] 人工抽检：重点抽查口径题、多对多题、时间语义题

### 5. 评测集分层策略

- [ ] 快速评测：首轮回归集（12 题）
- [ ] 标准评测：功能题 + 风险题全量跑
- [ ] 全面评测：加入 LLM Judge、Win Rate、人工抽检

### 6. 当前项目最值得优先落地的事项

- [ ] 为题库补充“标准校验 SQL”字段
- [ ] 增加自动化评测脚本，自动调用 `/api/stream/search-lite`
- [ ] 自动抽取每题的 stage、SQL、结果、错误信息
- [ ] 自动生成首版 Markdown 评测报告

### 7. 设计原则记录

- [ ] 不直接比较 SQL 字符串，优先比较结果集
- [ ] 结构化任务优先使用规则评测
- [ ] 非结构化输出再引入 LLM Judge
- [ ] 评测要支持版本对比，而不只是单次跑通
